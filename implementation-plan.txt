# Reliable Equipment Rentals — Service Dispatch Tool: Full Implementation Plan

## Context

Reliable Equipment Rentals Ltd needs a real-time Service Dispatch Board to replace their manual Excel + text message workflow. Daniel (dispatcher/operations lead) coordinates ~25 service calls/day across 5 field technicians servicing aerial lift equipment across BC's Lower Mainland. The current process is scattered across texts, phone calls, and a color-coded Excel spreadsheet — no single source of truth, no structured history, and delays when Daniel is on the phone.

Goal: A lightweight, real-time web app with two primary interfaces:
1. Dispatcher Dashboard (desktop-first) — weekly grid for managing all service calls
2. Technician Mobile View (mobile-first) — daily schedule with tap-to-navigate, tap-to-call

Tech Stack: Next.js 14+ (App Router), Convex (real-time backend), Convex Auth, Tailwind CSS + shadcn/ui, Vercel hosting.

## 1.4 — Convex Schema (convex/schema.ts)

5 tables plus Convex auth tables:

  Table           | Key Fields                                              | Indexes
  ----------------|--------------------------------------------------------|------------------------------------------
  users           | name, email, role (dispatcher|technician|admin),        | email, by_role
                  | technicianId?                                           |
  technicians     | name, phone, email, isActive, color                     | by_name, by_active
  serviceCalls    | rNumber, status (unassigned|assigned|in_progress|       | by_rNumber, by_status,
                  | on_hold|completed), customerName, contactName,          | by_assignedTechnician,
                  | contactPhone, machineInfo, complaint, address, city,    | by_scheduledDate,
                  | assignedTechnician?, scheduledDate?, callOrder?,        | by_technician_and_date (compound),
                  | priority?, dateOpened, requiresReturn, etc.             | by_status_and_technician (compound)
  callNotes       | serviceCallId, authorId, authorName, content,           | by_serviceCall, by_author
                  | noteType (general|parts_needed|return_required|         |
                  | swap|status_update)                                     |
  csvImports      | fileName, importedAt, importedBy, rowCount, status,     | by_importedBy, by_status
                  | errors?                                                 |

Key design decisions:
- scheduledDate as ISO string — calendar-day semantics, sorts correctly in indexes
- Compound index by_technician_and_date enables the core dispatcher grid query
- role directly on users table — simple union, avoids extra join on every auth check

## 1.6 — Convex Functions

  convex/serviceCalls.ts
    - list query — filters by status, technician, scheduledDate using compound indexes
    - getById query — single call with all fields
    - create mutation — defaults to status: "unassigned", requiresReturn: false
    - update mutation — partial field updates
    - assign mutation — sets technician + date + callOrder, status -> "assigned"
    - updateStatus mutation — validates allowed state transitions
    - unassign mutation — clears technician, status -> "unassigned"

  convex/callNotes.ts
    - listByServiceCall query — notes for a call ordered by _creationTime
    - create mutation — auto-fills authorId/authorName from auth context

  convex/technicians.ts
    - list query — all or active-only
    - getById query
    - create / update mutations
    - deactivate mutation — sets isActive: false

  convex/users.ts
    - currentUser query — gets authenticated user with role
    - setUserRole mutation — admin-only role assignment

## 4.4 — Client-Side Parser (src/lib/parsePointOfRentalExport.ts)

  Parses the uploaded file using xlsx (SheetJS) library. Runs client-side for instant preview.

  Expected POR columns:
    Status | Contract | Date Open | Due Date | Total | Owed | Item Name | Description | Location | Contact | Phone | Address | City | Zip | Operator Assigned

  Mapping logic:

    POR Column        | Our Field           | Transform
    -------------------|---------------------|------------------------------------------
    Contract           | rNumber             | Trim, normalize (ensure "r" prefix lowercase)
    Item Name          | itemName            | Direct
    Description        | machineInfo +       | Parse pattern: "Service Call - {model} -
                       | complaint           | #{unit} - {customer}"
    Location           | customerName /      | Direct (may need splitting)
                       | locationName        |
    Contact            | contactName         | Direct
    Phone              | contactPhone        | Direct
    Address            | address             | Direct
    City               | city                | Direct
    Zip                | postalCode          | Direct
    Operator Assigned  | assignedTechnician  | Fuzzy name match against technicians table
    Date Open          | dateOpened          | Parse date string to ISO

  Description field parser (src/lib/parseDescription.ts):
    Handles messy patterns like:
      "Service Call - S65 - #4443 - Midland"    -> model: "S65", unit: "4443"
      "Service Call - #8435 - 1930 - Keystone"  -> model: "1930", unit: "8435"
      "Service Call - S45 - #0185- Homecraft"   -> model: "S45", unit: "0185"
    Regex-based with fallback — extracts what it can, puts rest in complaint.

## 4.5 — Preview Table (src/components/import/ImportPreview.tsx)

# Architectural Decisions (All Phases)

  Decision                              | Rationale
  --------------------------------------|--------------------------------------------------
  Route groups (auth) / (dashboard)     | Different layouts without affecting URLs
  useQuery everywhere (no fetchQuery)   | Real-time subscriptions are the killer feature
  scheduledDate as ISO string           | Calendar-day semantics, sorts correctly in indexes
  Compound indexes                      | by_technician_and_date enables O(1) grid queries
  Undefined role = dispatcher           | Simplifies initial setup
  Client-side CSV parsing               | Instant preview without round-trip
  @dnd-kit for drag-and-drop            | Accessible, React 18+ compatible, lightweight
  localStorage for "last seen" notes    | Avoids extra table complexity
  Soft-delete technicians               | Preserves history
